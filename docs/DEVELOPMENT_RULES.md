# 開発・コーディング時の鉄則

## 進め方

1. 仕様を先に書く
   - 入力、出力、保存先、エラー時の結果を先に整理する。
   - 境界値を最低1つずつ決める。

2. 小さく作ってすぐ確認する
   - 1回の変更は「1画面」「1保存処理」「1帳票項目」など小さい単位にする。
   - 変更後は `pnpm run test`、`pnpm exec tsc --noEmit`、必要なら `pnpm run build` を通す。

3. 読める名前にする
   - 変数名は `data` や `tmp` だけで終わらせない。
   - 現場用語に寄せる。例: `calendarSchedules`, `workLogDate`, `invoiceNumber`。

4. 複雑な処理は共通部品に逃がす
   - HTMLエスケープ、PDF入力の正規化、ファイル制限などは `lib/` に置く。
   - 画面ファイルに長い安全処理を書き散らかさない。

5. テストを先に増やす
   - バグを直す時は、再発防止のテストを1つ追加する。
   - 境界値、空欄、長すぎる文字、危ない文字列を必ず含める。

## 自動チェック

- `pnpm run test`: 単体テスト
- `pnpm exec tsc --noEmit`: 型チェック
- `pnpm run build`: 本番ビルド確認
- `pnpm run check`: 上記をまとめて実行

## 変更前チェックリスト

- 入力値の上限はあるか
- 空欄の時に落ちないか
- 保存前に型や形式を整えているか
- 画面表示や帳票出力でHTMLを無害化しているか
- 既存データを壊さないか
- テストを追加または更新したか
